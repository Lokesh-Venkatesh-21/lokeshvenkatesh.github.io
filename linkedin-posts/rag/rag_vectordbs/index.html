05 — Going Deeper

Vector
Databases

A vector database isn't just storage — it's an Approximate Nearest Neighbour search engine. The algorithms it uses determine whether retrieval takes 5ms or 5 seconds at scale.

10 min readHNSW vs IVF6-database comparisonCode example

Why exact search doesn't scale

Finding the most similar vector to a query naively means comparing against every stored vector — O(n) operations per query. At 1 million chunks × 1536 dimensions, that's 1.5 billion floating-point operations per query. At 100 queries/second, the math becomes impossible.

Approximate Nearest Neighbour (ANN) algorithms solve this by sacrificing a tiny accuracy margin — finding top-10 results correctly 97% of the time instead of 100% — in exchange for sub-100ms search across tens of millions of vectors. In practice, the small recall loss is invisible: you're not looking for mathematical perfection, you're looking for chunks that give the LLM enough context.

HNSW vs IVF — the two main algorithms

HNSW (Hierarchical Navigable Small World)

Builds a multi-layer graph where each node connects to its nearest neighbours. At query time, navigation starts at the top (sparse, long-range connections) and descends to the bottom (dense, fine-grained neighbours). Excellent recall (95–99%) at very low latency (5–15ms at millions of vectors). The tradeoff: the entire graph must fit in RAM. Used by Weaviate, Qdrant, and pgvector.

HNSW — multi-layer graph navigation
Query enters top sparse layer and navigates down to dense result layer
▶ Run — query enters top layer → greedy navigation down → reaches nearest result

IVF (Inverted File Index)

Clusters vectors with k-means. At query time, only the nearest clusters are searched — a fraction of the total index. Far more memory-efficient than HNSW for very large indexes. Slightly lower recall if the correct result is in an unsearched cluster. Combined with Product Quantisation (PQ) as IVF-PQ for billion-scale search. FAISS uses IVF.

Index parameters that actually matter

HNSW: M and ef_construction

M — bidirectional connections per node. Default 16; try 32–64 for retrieval-critical applications. Higher M = better recall, more memory. ef_construction — candidate pool size during graph building. Higher = better graph quality, slower build. Don't reduce this to save build time and wonder later why recall is poor.

ef_search — the runtime recall lever

ef_search controls how many candidates are explored at query time. Must be ≥ top_k. Going from ef=64 to ef=128 typically adds 2–4ms but can recover 3–5% of missed relevant chunks — worth it in most RAG systems.

pgvector — HNSW index with tuned parameters
-- Create HNSW index CREATE INDEX ON chunks USING hnsw (embedding vector_cosine_ops) WITH (m = 32, ef_construction = 128); -- Set ef_search at query time for higher recall SET hnsw.ef_search = 128; -- Retrieve top-5 most similar chunks SELECT id, content, embedding <=> $1 AS distance FROM chunks ORDER BY embedding <=> $1 LIMIT 5;
