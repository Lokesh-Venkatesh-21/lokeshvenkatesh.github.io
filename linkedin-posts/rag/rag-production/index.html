07 — Going Deeper

Production
RAG

Getting RAG to work in a notebook is easy. Getting it to run reliably, cheaply, and fast in production is the hard part. Latency budgets, cost levers, semantic caching, and the failure modes you'll hit at scale.

The latency budget

A production RAG request has multiple serial latency contributors. Budget them explicitly — don't treat total latency as a black box and hope it's fast enough.

StepTypical latencyBottleneck riskLever
Query embedding20–80msLowSmaller model, cache identical queries
Vector search5–30msLow (HNSW)Reduce ef_search, use IVF at scale
Reranker (optional)80–300msMediumOnly rerank top-20, not top-100
LLM generation500–5000msHighStreaming, smaller model, reduce K
Streaming is your best latency lever. Time-to-first-token (TTFT) of 300ms feels fast even if total latency is 3s. Prioritise streaming over micro-optimising retrieval latency.

Cost optimisation

The three cost drivers

1. Embedding API calls. Charged per token. At scale, re-embedding identical or near-identical queries repeatedly adds up. Cache embeddings for common queries using a semantic cache — embed the query, check if a similar query exists in a fast in-memory store, return cached result.

2. LLM input tokens. K=10 chunks × 512 tokens each = 5,120 tokens just in context, every request. Reduce K aggressively — use a reranker to keep precision high at K=3–5. Compress retrieved context before passing to the LLM.

3. LLM output tokens. Prompt-engineer for concise answers. Route simple queries to a smaller, cheaper model — not everything needs GPT-4.

Python — semantic cache pattern
import numpy as np class SemanticCache: def __init__(self, threshold=0.95): self.cache = [] # list of (embedding, response) self.threshold = threshold def get(self, query_emb): for emb, resp in self.cache: if np.dot(query_emb, emb) >= self.threshold: return resp # cache hit — same semantic query return None # cache miss def set(self, query_emb, response): self.cache.append((query_emb, response))

Failure modes at scale

Context window overflow
K too high. Retrieved chunks + system prompt + conversation history exceeds the LLM's context limit. Model truncates silently or throws an error.
Track token counts explicitly. Set hard K limits per model. Monitor p95 context lengths in production. Use context compression.
Stale index
Vector index isn't updated when documents change. Retrieving outdated chunks that no longer reflect current truth — silent and dangerous.
Build incremental re-indexing into your ingestion pipeline. Version documents with timestamps. Monitor document freshness as an operational metric.
Embedding model drift
Embedding model is upgraded or swapped. Old vectors and new queries are in different spaces — similarity scores become meaningless. Quality silently collapses.
Never swap embedding models without re-indexing the entire corpus. Treat the embedding model as a hard dependency of your index version.
Lost in the middle
LLM attends poorly to context in the middle of a long prompt. Most relevant chunks placed in the middle of the context window are effectively ignored.
Put the most relevant chunks first and last. Use a reranker. Limit K so your context stays tight. Explicitly instruct the model to ground its answer in provided context.

Observability — what to monitor

Standard API latency monitoring isn't enough. RAG systems need pipeline-specific signals:

  • Retrieval latency p50/p95 — tracked separately from generation latency
  • Average similarity score of top-K chunks — sudden drops signal index or embedding issues
  • Context token count distribution — alert if p95 approaches your model's context limit
  • Faithfulness score — sample 5% of requests and run RAGAS faithfulness asynchronously
  • User feedback signals — thumbs down rate, regeneration rate, session abandonment after a response
  • Cache hit rate — track how often your semantic cache saves a round-trip
Tools: LangSmith, Arize Phoenix, Helicone, and Traceloop provide RAG-specific observability dashboards. LangSmith integrates directly with LangChain traces and shows retrieval quality per request.
