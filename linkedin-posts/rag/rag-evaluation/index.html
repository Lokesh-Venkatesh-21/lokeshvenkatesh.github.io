Evaluation

You can't improve what you can't measure. RAG evaluation requires measuring retrieval quality and generation quality separately — they fail differently and the fixes don't overlap.

10 min read4 core metricsRAGAS frameworkDiagnostic guide

Two failure modes, two evaluation axes

RAG systems fail in two fundamentally different ways. The fix for each is completely different — and confusing them wastes engineering time.

Retrieval failure: the right chunks weren't retrieved. The LLM had nothing useful — it generated from memory or hallucinated. Fix: chunking, embedding model, K value, reranker, index configuration.

Generation failure: the right chunks were retrieved but the LLM didn't use them correctly. It ignored them, misread them, or hallucinated on top of accurate context. Fix: system prompt, context ordering, LLM choice, context length management.

Diagnose first, fix second. Run both metric classes on 50–100 real queries. If context recall is 0.95 but faithfulness is 0.55 — your problem is generation, not retrieval. Swapping embedding models won't help.

The four core metrics

Faithfulness
Generation metric
supported_claims / total_claims_in_answer
Measures whether every factual claim in the answer is backed by the retrieved context. Score of 1.0 = nothing hallucinated. Low faithfulness: LLM is generating from memory or augmenting context with invented facts. Fix: stronger system prompt, better LLM, reduce context length.
Answer Relevancy
Generation metric
similarity(generated_answer, question)
Measures whether the answer addresses the actual question — not just whether it's factually grounded. A faithful but off-topic answer scores low here. Low relevancy usually indicates a query understanding problem — wrong context retrieved, or LLM misread the question. Fix: query rewriting, better embedding.
Context Recall
Retrieval metric
relevant_chunks_retrieved / all_relevant_chunks
Of all chunks in your corpus needed to answer the question, how many did retrieval actually surface? Low context recall means relevant content exists in your data but isn't being found. Fix: better embedding model, increase K, switch to Hybrid RAG, improve chunking strategy.
Context Precision
Retrieval metric
relevant_chunks_in_top_K / K
Of the K chunks retrieved, what fraction were actually useful? High recall + low precision floods the LLM with noise — at long context lengths, the "lost in the middle" phenomenon degrades accuracy. Fix: add a cross-encoder reranker, reduce K, tighten similarity threshold.

RAGAS — automated evaluation in practice

RAGAS computes all four metrics using an LLM-as-judge approach — a separate evaluator model (usually GPT-4) reads each example and scores it. This makes evaluation possible without expensive human labelling, though it introduces model-dependent biases. Always cross-validate against 20–30 human-labelled examples before trusting RAGAS scores for architecture decisions.

Python — RAGAS evaluation
from ragas import evaluate from ragas.metrics import faithfulness, answer_relevancy, context_recall, context_precision from datasets import Dataset data = { "question": ["What causes type 2 diabetes?", ...], "answer": ["Type 2 diabetes is caused by...", ...], "contexts": [["chunk1", "chunk2"], ...], "ground_truth": ["Insulin resistance and...", ...], } results = evaluate(Dataset.from_dict(data), metrics=[faithfulness, answer_relevancy, context_recall, context_precision]) print(results.to_pandas())
